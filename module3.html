1. Abstract Data Types (ADT)

An Abstract Data Type (ADT) is a mathematical model that defines a data type by its behavior from the user's perspective - specifically what operations can be performed and what those operations do. It focuses on what rather than how, hiding implementation details from users.

What Makes an ADT?

Data

The values the type can hold

Example: Stack stores elements

Operations

Actions that can be performed

Example: push(), pop(), peek()

Common ADT Examples

Stack ADT

LIFO

Last In, First Out (LIFO) collection

push(item)
pop()
peek()
isEmpty()

Queue ADT

FIFO

First In, First Out (FIFO) collection

enqueue(item)
dequeue()
front()
isEmpty()

List ADT

Ordered

Ordered collection with indexed access

add(item)
remove(index)
get(index)
size()

Key Insight

ADTs separate "what" from "how". Users know what operations are available but don't need to know how they're implemented. This is like using a TV remote - you know what the buttons do, but not the internal electronics!

2. Concept of ADTs & Data Abstraction

Data Abstraction is the process of hiding implementation details while exposing only essential features to users. It reduces complexity by focusing on what an object does rather than how it achieves it. This is a fundamental principle in software engineering.

Levels of Abstraction

1

Physical Level (Lowest)

How data is actually stored in memory (bits, bytes, pointers)

Memory addresses: 0x7fff5fbff8a0
2

Logical Level (Middle)

Data structures and how they organize data (arrays, linked lists)

Node → Node → Node → null
3

Abstract Level (Highest)

User interface - operations available to users (push, pop, enqueue)

stack.push(5), stack.pop()

Benefits of Data Abstraction

Security & Integrity

Prevents unauthorized access and manipulation of internal data

Reduced Complexity

Users don't need to understand implementation details

Reusability

Same ADT can be reused across different projects

Maintainability

Implementation can change without affecting users

Real-World Analogy: Driving a Car

✓ What You See (Abstract)

  • • Steering wheel
  • • Brake and gas pedals
  • • Gear shift
  • • Dashboard controls

✗ What's Hidden (Implementation)

  • • Engine mechanics
  • • Fuel injection system
  • • Transmission gears
  • • Electronic circuits

You can drive any car without knowing how the engine works - that's abstraction!

Remember

Good abstraction shows the minimum necessary information while hiding maximum complexity. It's about creating the right interface for users!

3. Encapsulation

Encapsulation is the bundling of data and methods that operate on that data within a single unit (class or object), while restricting direct access to some components. It's like putting valuable items in a safe - you control access through specific methods.

Core Principles

Bundling

Data and methods together in one unit

Data Hiding

Internal state hidden from outside

Controlled Access

Access through public methods only

Access Modifiers

public Accessible Everywhere

Can be accessed from any class

protected Subclass Access

Accessible within class and subclasses

private Restricted

Accessible only within the same class

Encapsulation Example

Without Encapsulation

class BankAccount {
  public double balance;
}

// Anyone can modify!
account.balance = -1000;

❌ Direct access allows invalid data

With Encapsulation

class BankAccount {
  private double balance;
  public void deposit(amt) {
    if (amt > 0)
      balance += amt;
  }
}

✓ Controlled access ensures validity

Best Practice

Make data members private and provide public getter/setter methods. This allows you to add validation, logging, or change implementation without affecting external code!

4. Interfaces vs Implementations

Understanding the distinction between interfaces (what) and implementations (how) is crucial for effective software design. The interface defines a contract, while implementation provides the actual working code.

Interface

The Contract

Definition

Specifies what operations are available

Focus

Method signatures, behavior specification

User View

What clients see and interact with

Changes

Rarely changes (affects all users)

Implementation

The Code

Definition

Specifies how operations work internally

Focus

Algorithms, data structures, logic

User View

Hidden from clients

Changes

Can change freely without affecting users

Practical Example: List ADT

List ADT - Multiple Implementations

Interface (What)

add(item), remove(index), get(index), size()

Same operations for all implementations

Implementation 1: Array-Based

  • ✓ Fast random access O(1)
  • ✗ Slow insertion O(n)
  • ✓ Memory efficient

Implementation 2: Linked List

  • ✗ Slow random access O(n)
  • ✓ Fast insertion O(1)
  • ✗ Extra memory for pointers

Why Separation Matters

Flexibility

Change implementation without affecting client code

Team Collaboration

Different teams can work on interface and implementation separately

Multiple Implementations

One interface, many implementations optimized for different use cases

Testing

Test interface contract separately from implementation details

Design Principle

"Program to an interface, not an implementation" - This famous principle allows your code to be flexible, maintainable, and testable. Always depend on abstractions, not concrete implementations!
