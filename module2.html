1. Time Complexity

Time complexity measures the computational time an algorithm requires as a function of input size. It quantifies how the runtime scales when processing larger datasets, helping developers predict performance and identify bottlenecks.

Why Time Complexity Matters

Performance

Predicts how fast algorithms run on different input sizes

Scalability

Shows how algorithms behave as data grows

Comparison

Enables objective algorithm comparison

Common Time Complexities

O(1)

Constant Time

Execution time doesn't change with input size

Example: Array access, hash table lookup

O(log n)

Logarithmic Time

Grows slowly, divides problem in half repeatedly

Example: Binary search, balanced tree operations

O(n)

Linear Time

Grows proportionally with input size

Example: Linear search, iterating through array

O(n log n)

Linearithmic Time

Efficient sorting complexity

Example: Merge sort, quicksort (average)

O(n²)

Quadratic Time

Time grows with square of input

Example: Bubble sort, nested loops

O(2ⁿ)

Exponential Time

Doubles with each additional input

Example: Recursive fibonacci, subset generation

Remember

Time complexity focuses on growth rate, not exact time. An O(n) algorithm isn't necessarily slower than O(1) for small inputs - constants matter in practice!

2. Space Complexity

Space complexity quantifies the total memory an algorithm requires during execution. This includes memory for input data, auxiliary variables, data structures, and recursive call stacks. Like time complexity, it's expressed using Big O notation.

Components of Space Complexity

Fixed Space

Space required for code, constants, and simple variables

Independent of input size

Variable Space

Space for data structures, recursion, temporary variables

Depends on input size

Space Complexity Examples

Simple Variable Storage

O(1)
int sum = 0;
for (int i = 0; i < n; i++) {
  sum += arr[i];
}

Uses only a fixed amount of extra space regardless of input size

Array Copy

O(n)
int[] copy = new int[n];
for (int i = 0; i < n; i++) {
  copy[i] = arr[i];
}

Space grows linearly with input size due to new array

Recursive Factorial

O(n)
int factorial(int n) {
  if (n <= 1) return 1;
  return n * factorial(n - 1);
}

Call stack grows with recursion depth (n calls)

Space-Time Tradeoff

Often you can reduce time complexity by using more space (caching, memoization) or reduce space by accepting slower algorithms. Choose based on your constraints!

3. Growth of Functions

The growth of functions describes how algorithm runtime or space requirements increase as input size grows. Understanding growth rates helps predict which algorithm will perform better for large inputs, even if constants make one slower for small datasets.

Comparing Growth Rates

Complexity n=10 n=100 n=1,000 n=10,000
O(1) 1 1 1 1
O(log n) 3 7 10 13
O(n) 10 100 1,000 10,000
O(n log n) 30 664 9,966 132,877
O(n²) 100 10,000 1,000,000 100,000,000
O(2ⁿ) 1,024 1.27×10³⁰ ∞ ∞

Key Insight

For large inputs, O(log n) is vastly better than O(n), and O(n) is much better than O(n²). The difference between O(n) and O(n²) for n=10,000 is 10,000 vs 100,000,000 operations!

4. Big-O Notation

Big-O notation describes the upper bound of an algorithm's growth rate. It represents the worst-case scenario - the maximum time or space an algorithm will ever need. Big-O is the most commonly used notation for analyzing algorithms.

Formal Definition

A function f(n) is O(g(n)) if there exist positive constants c and n₀ such that:

0 ≤ f(n) ≤ c · g(n) for all n ≥ n₀

Translation: f(n) grows no faster than g(n) times some constant, for sufficiently large n

Big-O Rules

1

Drop Constants

O(2n) → O(n), O(5n²) → O(n²)

2

Drop Lower-Order Terms

O(n² + n) → O(n²), O(n³ + n² + n) → O(n³)

3

Keep Dominant Term

Focus on the term that grows fastest

4

Different Inputs → Different Variables

Two arrays of different sizes: O(a + b) not O(n)

Practical Examples

Array Element Access

O(1)
int value = arr[5]; // Direct access

Linear Search

O(n)
for (int i = 0; i < n; i++) {
  if (arr[i] == target) return i;
}

Nested Loops

O(n²)
for (int i = 0; i < n; i++) {
  for (int j = 0; j < n; j++) {
    // operation
  }
}

5. Big-Theta (Θ) Notation

Big-Theta notation describes the tight bound of an algorithm's growth rate. It represents both the upper and lower bounds simultaneously - meaning the algorithm's complexity grows at exactly this rate. Theta gives the most precise description of complexity.

Formal Definition

A function f(n) is Θ(g(n)) if there exist positive constants c₁, c₂, and n₀ such that:

0 ≤ c₁ · g(n) ≤ f(n) ≤ c₂ · g(n) for all n ≥ n₀

Translation: f(n) is sandwiched between two multiples of g(n)

O

Upper Bound

≤ (At most)

Θ

Tight Bound

= (Exactly)

Ω

Lower Bound

≥ (At least)

When to Use Theta

Loop Through Array

Always visits every element exactly once

Θ(n)

Nested Loop (NxN)

Always performs n² iterations

Θ(n²)

Binary Search

May find element early (not tight bound)

O(log n) not Θ(log n)

Important Note

Theta is more precise than Big-O, but we can only use it when best and worst cases have the same complexity. Use Big-O when in doubt!

6. Big-Omega (Ω) Notation

Big-Omega notation describes the lower bound of an algorithm's growth rate. It represents the best-case scenario - the minimum time or space an algorithm will need. While less commonly used than Big-O, it's important for understanding algorithm performance range.

Formal Definition

A function f(n) is Ω(g(n)) if there exist positive constants c and n₀ such that:

0 ≤ c · g(n) ≤ f(n) for all n ≥ n₀

Translation: f(n) grows at least as fast as g(n) times some constant

Notation Relationships

O(g(n)) f(n) grows no faster than g(n)
Ω(g(n)) f(n) grows no slower than g(n)
Θ(g(n)) f(n) grows at the same rate as g(n)

Practical Examples

Linear Search (Best Case)

Ω(1)

Element found at first position

Best: [5, 3, 8, 1] searching for 5 → found immediately!

Merge Sort

Ω(n log n)

Even in best case, must divide and merge all elements

Bubble Sort (Best Case)

Ω(n)

Already sorted array - one pass to confirm

Remember

If an algorithm is Θ(f(n)), then it is also O(f(n)) AND Ω(f(n)). Theta means both upper and lower bounds match!

7. Best, Average, and Worst Cases

Algorithm performance varies based on input characteristics. Understanding best, average, and worst-case scenarios helps you predict real-world performance and make informed decisions about which algorithm to use.

Best Case

Minimum time/space needed. Occurs with most favorable input.

Notation:

Ω (Omega)

Average Case

Expected time/space for typical inputs. Most realistic measure.

Notation:

Θ (Theta)

Worst Case

Maximum time/space needed. Occurs with least favorable input.

Notation:

O (Big-O)

Real Algorithm Examples

Linear Search

Best Case: Ω(1)

Element at first position

Average Case: Θ(n)

Element in middle on average

Worst Case: O(n)

Element at last or not found

for (int i = 0; i < n; i++) {
  if (arr[i] == target) return i; // Found!
}
return -1; // Not found

Binary Search

Best Case: Ω(1)

Element at middle position

Average Case: Θ(log n)

Typical search depth

Worst Case: O(log n)

Maximum tree depth

Note: Requires sorted array

Quick Sort

Best Case: Ω(n log n)

Balanced partitions

Average Case: Θ(n log n)

Typical random pivot

Worst Case: O(n²)

Already sorted, bad pivot

Note: Worst case rare with random pivot selection

Common Algorithm Complexities

Algorithm Best Average Worst
Linear Search Ω(1) Θ(n) O(n)
Binary Search Ω(1) Θ(log n) O(log n)
Bubble Sort Ω(n) Θ(n²) O(n²)
Quick Sort Ω(n log n) Θ(n log n) O(n²)
Merge Sort Ω(n log n) Θ(n log n) O(n log n)
Heap Sort Ω(n log n) Θ(n log n) O(n log n)

Design Decision

In practice, we often focus on worst-case (Big-O) because it guarantees performance won't exceed that bound. Average case is valuable when typical inputs differ significantly from worst case.
